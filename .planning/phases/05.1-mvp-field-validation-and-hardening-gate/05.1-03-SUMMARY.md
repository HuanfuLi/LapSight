# Phase 5.1 Plan 03: Ready Gate, Raw-Recording Seam, and Source Provenance Fix Summary

**A single conservative pure `aggregateReady(...)` gate over all seven D-14 inputs plus wrong-course preflight, a distinct diagnostic `RawRecordingController` seam that builds no lap/ghost state and is structurally excluded from valid sessions, the confirmed P1 evidence-integrity fix so real runs carry the live PhoneGps source, and Ready + raw-recording state surfaced on the Drive dash — all clean-room, unit-covered, with the full host suite and `:androidApp:assembleDebug` green.**

## Chosen Ready Thresholds (for transcription into 5.1-UAT.md by Plan 06)

| Input | Threshold | Source / rationale |
|-------|-----------|--------------------|
| Horizontal accuracy | `maxHorizontalAccuracyMeters = 25.0` | The engine's degraded-accuracy line (`GpsQualitySummary.DEFAULT_DEGRADED_ACCURACY_METERS`), stricter than preflight's 100 m course-distance gate. |
| Fix freshness | `maxFixAgeMillis = 15_000` | The wrong-course preflight freshness window (15 s). |
| Sample rate / cadence | `minSampleRateHz = 1.0` | The observed real Fused-Location delivery floor (RESEARCH Nyquist note: requested ~10 Hz, real delivery often 1 Hz). Below this a start/finish crossing cannot be trustworthily reconstructed. |

All thresholds are injectable via `ReadyThresholds` and validated by `init { require(...) }`.

## Accomplishments

- **Task 1 — Pure aggregateReady gate (ReadyGate.kt):** A clean-room `commonMain` decision function (zero Compose/platform imports) over fix presence/validity, horizontal accuracy, freshness, sample rate/cadence, course/profile selection, confirmed start/finish, direction compatibility, and wrong-course preflight. Returns a sealed `ReadyState.Ready` / `NotReady(reasons: List<ReadyBlocker>)` (never a bare Boolean). Motion/speed/heading are NOT inputs (D-15). Non-finite/out-of-range GPS returns `NotReady(NonFiniteFix)`, never an exception (T-5.1-02). `ReadyThresholds` validates non-finite/non-positive values at construction. `ReadyGateTest` covers one case per `ReadyBlocker`, the all-satisfied Ready case, a multi-reason case, threshold validation, and the bad-input-is-data contract.
- **Task 2 — Raw-recording seam (RawRecordingController.kt):** A thin diagnostic seam mirroring the `DriveMarkingController` provider/tick/stop loop but producing a `RawRecordingPayload` (samples via `LocationSample.toDto()` + `GpsQualitySummary.from(samples)`) flagged `isDiagnosticRaw = true` with `lapCount = 0`. It constructs NO `TimingSessionRecorder`/`LapEngine`/`LiveDeltaEngine`, so its output is structurally excluded from valid-session counting (D-16/D-17). `RawRecordingControllerTest` proves samples accumulate, the snapshot exposes a GPS-quality rollup, tick is a no-op before start, and the stop payload carries the raw marker with zero laps.
- **Task 3 — Gating + provenance + dash state:**
  - `SessionController.startTiming` gains `recentRateHz` + opt-in `requireReady`; when `requireReady` is set it computes `aggregateReady(...)` after the wrong-course check and returns the new `StartTimingResult.NotReady(reasons, message)` instead of starting formal timing (D-13). The existing `WrongCoursePreflight` Blocked/override path is preserved (D-18); `overrideWrongCourseAndStart` intentionally bypasses the Ready gate to consume the captured evidence.
  - `AppShell` now constructs `SessionController` with a `sourceForTrack` override that injects the live feed source (`PhoneGps` for the phone provider, `Simulated` for the demo feed) via `rememberUpdatedState`, so a real run on any Track is labeled by the live feed, not the Track's marking source (D-04/D-42 — the confirmed P1).
  - `DriveScreen` surfaces a Ready/not-Ready glance indicator (with a primary not-Ready reason) and a raw-recording-active indicator in the status bar, drives a `RawRecordingController` over the live feed, and exposes a "Record raw GPS (diagnostic)" path when not Ready plus a "Stop raw recording" control while recording (D-16/D-32). The manual orientation toggle is untouched; no sensor rotation introduced.

## Decisions Made

- **Conservative thresholds derived from existing constants** (25 m / 15 s / 1.0 Hz) so Ready is strict but reachable; recorded above for Plan 06.
- **Opt-in Ready gate** (`requireReady=false` default): preserves the existing `startTiming` contract for the ~30 determinism/integration call sites that drive the engine without a fresh GPS fix, while the production Drive UI passes `requireReady=true`.
- **Order: wrong-course Blocked → Ready gate.** The overridable wrong-course evidence path (D-18) must take precedence so the user can still override; the Ready gate then covers the remaining inputs plus a preflight that is `Unavailable`.
- **Dash Ready preview vs authoritative gate.** The dash computes a lightweight `dashReadyState` over the inputs it can see; the authoritative, complete gate (including freshness, direction, and preflight) runs inside `SessionController.startTiming`.
