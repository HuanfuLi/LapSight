## Summary

LapSight's Phase 1-5 code is a mature KMP/Compose Multiplatform mounted-phone lap timer with a genuinely clean-room, deterministic lap engine. The shared domain (`shared/src/commonMain/.../lap`, `ghost`, `session`, `track`, `storage`, `export`) is well separated from Compose/platform code, and the `LapEngine.onSample()` pipeline is pure with respect to its sample input — no `Random`, no wall-clock reads, no platform calls inside the algorithm. This is the single most important fact for this gate: **the determinism hard-gate (D-25..D-28) rests on a sound foundation, but the existing determinism test only asserts `LapEngine.finalState` equality across two in-memory runs.** It does not yet assert determinism across the full pipeline (sector results, ghost delta state transitions, course/direction decisions) nor "replay an exported JSON session" — there is no JSON-to-replay path at all. Building that replay-from-export harness and a full-pipeline determinism assertion is the central new test work of this phase.

The biggest hardening gap is **the conservative aggregate Ready state required by D-13/D-14**. Today there is no single "Ready" gate. `DriveMarkingController.snapshot().canStartTiming` only checks that a current Track profile is selected. `WrongCoursePreflight` evaluates GPS-fix validity, accuracy, freshness, and whole-course distance — but only at `startTiming()` time, only if `latestGps` is passed, and it does not check sample rate/cadence, and it is not surfaced as a pre-timing "Ready/not-Ready" indicator combining all seven D-14 inputs. There is also **no distinct "raw recording while not Ready" mode** (D-16/D-17): the app has Mark-New-Track capture and formal Timing, but no diagnostic raw-GPS recording that explicitly refuses to start lap/ghost state. These two are the most likely "Hardening Required" items and should be planned as the only feature-shaped work permitted in this gate.

License/privacy in core paths is mostly clean: no network, analytics, or third-party SDK in `commonMain`, and no copied Doves/GPL code found in `shared/src`. However ARCH-03 (document third-party code/licenses) and ARCH-04 (GPL-not-copied attestation) are not done and the app license is still undecided (STATE.md checklist) — these are legitimate audit findings under D-41/D-45.

**Primary recommendation:** Plan three workstreams — (1) a deterministic **replay-from-export harness** plus a full-pipeline N-times-identical determinism test suite spanning lap/sector/ghost/course-direction/recovery; (2) a conservative **aggregate Ready gate** and an explicit **raw-recording mode** (the minimum hardening features D-13..D-17 demand); (3) the audit + UAT + UI-audit + field-test evidence machinery (five artifacts). Treat the existing `GpsFixtureLibrary` scenarios as the deterministic sampling backbone and extend it to cover the GPS-quality/lap-geometry/direction/course-selection/recovery regimes the Nyquist VALIDATION.md will require.

### Pattern: Pure-engine + injected-clock determinism
**What:** Algorithm code (`LapEngine`, `LiveDeltaEngine`, `CourseProgressMatcher`, `WrongCoursePreflight`) takes only sample/geometry inputs; all wall-clock and ID generation is isolated behind injectable `now: () -> Long` in controllers (`SessionController`, `DriveMarkingController`, `TrackProfileController`).
**When to use:** Every determinism test — construct controllers with a fixed `now` and an `InMemorySessionStore`; feed fixture samples; assert structural equality.
**Example:**
```kotlin
// Source: shared/src/commonTest/.../lap/ReplayTest.kt (existing pattern)
@Test fun replayIsDeterministic() {
    val a = runner().run(ReplayFixtures.multiLapLoop())
    val b = runner().run(ReplayFixtures.multiLapLoop())
    assertEquals(a.finalState, b.finalState)   // ← only finalState today; widen this (see Pitfall 1)
}
```

### Anti-Patterns to Avoid
- **Asserting only `finalState`** for determinism — misses sector-result and ghost-delta state-transition drift (D-26). Assert the full step/result sequence.
- **Putting Ready logic in Compose** — D-14 must be a pure function so replay/deterministic tests can cover the discretionary thresholds (CONTEXT discretion clause).
- **Letting a simulated run masquerade as real** — `SessionController.sourceForTrack` defaults to the Track's marking source; for real field runs the production wiring must inject `LocationSource.PhoneGps`, or simulated isolation (D-42/D-43) breaks.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Replaying a captured field session | A second parser/engine | Decode the exported `TimingSessionPayloadV1` JSON → `LocationSample` list → existing `ReplayRunner`/`TimingSessionRecorder` | A parallel engine would itself need validating; reuse the clean-room engine (AGENTS.md). |

| Structural diff of replay outputs | Custom comparator | `data class` `equals` on `LapTimingState`/`SectorResult`/`LiveDeltaSnapshot` | Kotlin gives value equality for free. |
| Lap-count ground truth | Trusting the app | Independent video + manual tally (D-19/D-22) | The whole gate exists to catch the app being wrong. |

**Key insight:** The app under validation must never also be the validator. Ground truth for lap count comes from human tally + video; ground truth for determinism comes from re-running the *same* engine on the *same* bytes and demanding identical structured output.

## Common Pitfalls

### Pitfall 1: Determinism test scope is too narrow today
**What goes wrong:** `ReplayTest.replayIsDeterministic` asserts only `finalState`. D-26 demands lap count, lap times, **sector times, ghost-delta availability/state transitions, and course/profile/direction decisions** all be deterministic.
**Why it happens:** The existing test predates the ghost/sector/direction features.
**How to avoid:** Add a determinism suite that runs each fixture (and each decoded export) N times through the full `SessionController`+`TimingSessionRecorder` pipeline and asserts equality of: completed laps list, `completedSectorResults`, the ordered `LiveDeltaSnapshot` sequence, and the resolved `CourseCompatibilityKey`/direction.
**Warning signs:** A test that compares two runs but only checks `lapCount`.

### Pitfall 2: No JSON→replay path exists
**What goes wrong:** D-12/D-25/D-28 require replaying an *exported field session*; today `JsonExportService` only encodes, nothing decodes JSON back into samples for the engine.
**Why it happens:** Export was built for sharing/debug, not for a closed-loop replay harness.
**How to avoid:** Add a small decode helper (`TimingSessionPayloadV1` → `List<LocationSample>` via existing `*.toModel()` mappers) and a replay entry point that re-runs through the same course/direction. This is the canonical mechanism that links field evidence to deterministic validation.
**Warning signs:** A field-test plan that "replays" by re-driving rather than by re-feeding exported bytes.

### Pitfall 3: Ready is not a single conservative gate (D-13/D-14)
**What goes wrong:** `canStartTiming` only checks track selection; `WrongCoursePreflight` covers fix/accuracy/freshness/distance but **not sample rate/cadence** and is not surfaced as a unified Ready state. A session could "start timing" without the conservative Ready D-13 requires, invalidating it as evidence.
**Why it happens:** Ready was implicitly split across selection + preflight + engine filters.
**How to avoid:** Build one pure `aggregateReady(...)` returning Ready/NotReady(reasons) over all seven D-14 inputs (fix, accuracy, freshness, rate/cadence, course/profile, confirmed start/finish, direction compatibility) plus wrong-course preflight; gate formal timing on it; surface it on the dash (D-32).
**Warning signs:** Any field session where timing began while the dash showed a poor/acquiring GPS state.

### Pitfall 4: Raw recording vs formal timing not separated (D-16/D-17)
**What goes wrong:** When not Ready, the user must be able to record raw GPS for diagnosis **without** entering lap/ghost timing. No such mode exists; the only capture paths are Mark-New-Track and formal Timing.
**How to avoid:** Add a raw-recording seam that streams samples to storage/export but constructs no `LapEngine` lap state and no `LiveDeltaEngine` reference; mark its output clearly so it never counts toward the five valid sessions.
**Warning signs:** Diagnostic recordings appearing in Review as timing sessions with lap counts.

### Pitfall 5: Simulated data leaking into real evidence (D-42/D-43/D-04)
**What goes wrong:** `SessionController`'s default `sourceForTrack` returns the Track's marking source, so a real run on a Demo-created Track stays "Demo," and conversely production wiring must inject `PhoneGps`.
**How to avoid:** Audit the Android production wiring (`MainActivity` → controllers) to confirm the live provider's `LocationSource.PhoneGps` flows into the session source; verify field exports carry a real (non-simulated) source.
**Warning signs:** A field session export whose `source.isSimulated == true`.

### Pitfall 6: Floating-point cross-platform divergence
**What goes wrong:** Determinism is asserted on JVM/Android host; iOS is out of scope (D-02) but the engine uses `sin/cos/atan2/sqrt`. Within Android the math is reproducible; do not over-claim cross-platform identical bits.
**How to avoid:** Scope determinism claims to the Android validation target (matches D-02/D-04). Use exact integer-millis equality for times (the engine works in `Long` millis) and structural equality for states; reserve tolerances only for any derived doubles.
**Warning signs:** A determinism test that fails only on a different host than where fixtures were generated.

### Pitfall 7: UI audit already shows 3 sub-3 pillars (D-37 blocker)
**What goes wrong:** The existing `05.1-UI-REVIEW.md` scores Color 1/4, Typography 1/4, Spacing 2/4 (overall 13/24) — but uses a *different* rubric ("Copywriting/Visuals/Color/Typography/Spacing/Experience Design") than the **six D-36 pillars** (Domain Fit, Information Hierarchy, State Clarity, Motion/Glance Readability, Responsiveness/Device Fit, Visual Polish/Accessibility).
**How to avoid:** Re-run the UI audit against the exact D-36 pillar set. The hardcoded color/typography/spacing findings map into "Visual Polish/Accessibility" and would block Go (< 3/4) until tokenized. Plan a design-token hardening task (MaterialTheme color/type/spacing) so that pillar reaches >= 3/4.
**Warning signs:** Reusing the prior 6-axis scores as if they were the D-36 pillars.

### Pitfall 8: Orientation must stay manual (project memory)
**What goes wrong:** Any move to sensor/auto rotation for the mounted dash violates the recorded memory rule (racing G-forces flip the screen).
**How to avoid:** Confirm the activity orientation is locked and the layout branches on an explicit user toggle, not the sensor. Treat any `screenOrientation="sensor"/"fullSensor"` as a finding.

## Validation Architecture

> The plan-phase orchestrator consumes this section to generate the Nyquist VALIDATION.md. It is the heart of the gate. It composes five evidence layers — code audit, deterministic replay, on-device UAT, mounted-display UAT, and closed-course field tests — into one defensible Go / Hardening-Required / No-Go (D-50..D-53).

### Layers / Seams to Validate

1. **Provider seam** (`LocationSampleProvider`): real (`AndroidFusedLocationSampleProvider`) vs simulated (`SimulatedGpsProvider`) produce `LocationSample` identically; source metadata is honest (D-42/D-43/D-04). *Audit + UAT.*
2. **Ready gate seam** (NEW, D-13/D-14): pure aggregate Ready over fix/accuracy/freshness/rate/course/start-finish/direction/preflight. *Replay + UAT.*
3. **Raw-recording seam** (NEW, D-16/D-17): records samples without lap/ghost state; excluded from valid-session counting. *UAT + audit.*
4. **Lap engine seam** (`LapEngine`/`CrossingDetector`): crossing, direction gate, cooldown, min-lap, accuracy/speed filters, complete-Sector intervals. *Replay (primary).*
5. **Ghost seam** (`LiveDeltaEngine`/`CourseProgressMatcher`): match/unmatch transitions, suppression to `--`, in-session new-best reference update. *Replay.*
6. **Course-identity seam** (`CourseGeometryBuilder`/`GhostCompatibility`/`WrongCoursePreflight`): direction (Recorded/Reverse), revision compatibility key, wrong-course preflight + override evidence. *Replay + UAT.*
7. **Session lifecycle seam** (`SessionController`/`TimingSessionRecorder`/store): start/stop/save/discard/draft-recovery; checkpoint integrity. *Replay + UAT.*
8. **Evidence seam** (`JsonExportService` + NEW JSON→replay decode): export → replay → identical output; links field session to deterministic result (D-12/D-25/D-28). *Replay + field.*
9. **UI surface seam** (`DriveScreen`/`AppShell`): one-second-glance core state (D-31/D-32), six-pillar audit (D-36), manual orientation. *UI audit + display UAT.*

### Deterministic Replay Strategy & Nyquist Sampling Adequacy

The gate is trustworthy only if the **input space is sampled densely enough** that passing replay implies real-world correctness. Map each input dimension to fixtures; flag where current coverage is thin (Wave 0 work).

| Input dimension (regime) | Required samples | Existing fixtures | Gap → Wave 0 |
|--------------------------|------------------|-------------------|--------------|
| **GPS quality** | clean, noisy+drift, dropped/low-frequency, one-outlier-loop, poor-accuracy crossing | `CLEAN_10_LOOP`, `NOISE_DRIFT`, `DROPPED_LOW_FREQUENCY`, `ONE_OUTLIER_LOOP`, `poorAccuracyCrossing` | Add a **degradation-during-timing** scenario (good→poor mid-lap) to prove Ready/suppression (D-14/D-24). |
| **Lap geometry** | single lap, multi-lap, min-5-loop, jitter-near-line (no false lap), low-frequency lap | `oneLapLoop`,`multiLapLoop`,`MINIMUM_5_LOOP`,`jitterNearLine`,`lowFrequencyLap` | Add a **closed-course perpendicular start/finish** fixture mirroring the hardened Drive boundary (STATE: "perpendicular start/finish for closed-path captures"). |
| **Direction** | recorded forward, reverse, opposite first-crossing rejected, mid-lap turnaround | `CourseDirectionReplayTest` fixtures (in test) | Promote turnaround/reverse cases into the standing determinism suite, not just behavior tests. |
| **Course selection** | correct course Ready, pit/paddock near-course allowed, clearly-far blocked, accuracy-margin non-block, stale fix unavailable | `wrongCoursePitPaddock`,`wrongCourseFarCourse`,`wrongCourseAccuracyMargin`,`wrongCourseStale`,malformed/non-finite | Add **direction-incompatible selected ghost** case feeding aggregate Ready (D-14). |
| **Session recovery** | resume mid-draft, save-after-stop, discard-never-persists | `DraftRecoveryTest`, `SessionControllerTest` | Add **replay-equals-resume** determinism (resumed engine state == fresh replay of same samples). |
| **Ghost delta** | slower (+), faster (−), in-session new best, unmatched→`--`, recovery rematch, backward motion | `VARIABLE_PACE_GHOST_UAT`,`COURSE_MATCH_RECOVERY`,`COURSE_MATCH_BACKWARD`,`COURSE_MATCH_AMBIGUITY` | Add ghost **state-transition-sequence** determinism assertion (not just final delta). |

**Sampling-rate (literal Nyquist) note:** Phone Fused Location is requested at ~10 Hz (`UPDATE_INTERVAL_MILLIS=100`), but real delivery is often 1 Hz and the `DROPPED_LOW_FREQUENCY` fixture is 12 points/loop. A start/finish crossing event must be reconstructable from sub-sampled GPS — the engine already interpolates crossing time within a movement segment (LAP-03). Validation must include the **low-frequency regime** as a first-class case (it is the most likely real-world lap-count failure mode, D-19/D-24). Where a lap cannot be trustworthily reconstructed, the app must degrade/suppress (D-24), and replay must show that decision deterministically.
