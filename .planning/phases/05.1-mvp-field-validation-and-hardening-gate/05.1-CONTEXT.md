# Phase 5.1: MVP Field Validation and Hardening Gate - Context

**Gathered:** 2026-06-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 5.1 freezes feature expansion after Phase 5 and validates whether the Android phone app is trustworthy as a mounted-phone lap timing instrument before external GNSS and Meta glasses work begins.

This phase is an Android phone-only validation and hardening gate. It does not validate iOS, external GNSS devices, Meta glasses, watches, cloud features, public-road use, advanced telemetry charts, map ghost animation, or any other peripheral adaptation. It may add only the minimum instrumentation, replay tooling, tests, or fixes required to diagnose and harden the existing Phase 1-5 phone MVP.

The output is an Android MVP Go/No-Go decision. A Go decision means the Android phone timing state is trusted enough to begin Phase 6 and Phase 7 work under the limits captured below. It does not claim complete dual-platform validation.

</domain>

<decisions>
## Implementation Decisions

### Validation Level and Platform Scope

- **D-01:** Phase 5.1 uses a release-prep hard-gate mindset, not a light MVP smoke test.
- **D-02:** The validation scope is Android phone-only for this phase. iOS is explicitly not validated and remains a documented risk.
- **D-03:** No extra devices are tested in Phase 5.1. External GNSS, Meta glasses, watches, and other peripherals remain outside the validation scope.
- **D-04:** A successful Phase 5.1 creates an `Android phone MVP validated` conclusion, not a full multi-platform MVP conclusion.

### Android Device Matrix

- **D-05:** Use at least two real Android phones.
- **D-06:** One device is the primary Android phone. The second device is any other available Android phone; it does not need to meet a forced low-end, old-device, brand, chipset, or screen-size criterion.
- **D-07:** Record device identity for each field session, including at least device name/model when available and enough context to distinguish the two devices.
- **D-08:** Android emulators may be used for supplemental UI/layout smoke checks, but emulator results do not count as core field-validation evidence.

### Closed-Course Field Validation

- **D-09:** Closed-course testing is a hard requirement for Phase 5.1 Go. Walking or bike loops may be used for debugging and prep, but they do not replace closed-course validation.
- **D-10:** Phase 5.1 requires at least five complete valid closed-course sessions.
- **D-11:** The five closed-course sessions are distributed as three sessions on the primary Android phone and two sessions on the second Android phone.
- **D-12:** Each valid closed-course session must preserve enough evidence for replay and review: session export, replay output, field notes, and video reference.

### Effective Session and Ready Semantics

- **D-13:** A session is valid for accuracy and Go/No-Go only if the app itself showed the required Ready state before timing began.
- **D-14:** Ready is conservative. The app should show Ready only when GPS fix, horizontal accuracy, sample freshness, sample rate/update cadence, selected course/profile, confirmed start/finish, and course direction compatibility satisfy the implementation thresholds.
- **D-15:** The user does not need to already be moving for Ready. Motion stability, speed, and heading are not mandatory base Ready inputs.
- **D-16:** If the app is not Ready, the user may start raw GPS recording for diagnosis, but the app must not enter formal lap timing or ghost timing for that recording.
- **D-17:** Raw recording while not Ready does not count toward the five valid closed-course sessions.
- **D-18:** Clearly wrong-course preflight blocks Ready, consistent with Phase 5. The existing explicit `Still use this track` override remains allowed and must be recorded as session evidence visible in Review/export.

### Lap Count and Timing Accuracy

- **D-19:** Lap count is the hard field-validation standard. Every valid closed-course session must produce 100% correct lap count.
- **D-20:** Any valid closed-course session with an incorrect lap count is a Phase 5.1 No-Go for that validation run. The issue must be fixed and affected validation rerun before Phase 6 or Phase 7 can begin.
- **D-21:** Lap time accuracy target is `<= 0.5s` against video-derived crossing times for laps where the video clearly shows start/finish crossing. This is a target metric, not an automatic Phase 6 blocker.
- **D-22:** There is no dedicated professional timing device requirement. Video reference plus deterministic replay is the accepted workaround.
- **D-23:** If a video crossing is not clearly readable, that lap is excluded from timing-accuracy conclusions but may still support stability, storage, replay, UI, or lap-count evidence.
- **D-24:** Phone GPS limitations must be handled honestly. Interpolation and smoothing may improve crossing time, speed, and delta from discrete GPS samples and timestamps, but the app must suppress or degrade timing when the input data cannot support a trustworthy result.

### Replay and Algorithm Determinism

- **D-25:** Replay determinism is a hard gate. The same raw GPS/session export replayed repeatedly must produce identical algorithmic outputs.
- **D-26:** Deterministic outputs include lap count, lap times, sector times, ghost delta availability/state transitions, and course/profile/direction decisions.
- **D-27:** Differences are allowed only in display formatting or rounding. Algorithmic output drift is not allowed.
- **D-28:** Replay must be used to diagnose every field failure or timing anomaly before deciding whether the problem is data quality, line placement, interpolation, Ready classification, course selection, or implementation logic.

### Mounted-Phone Display UAT

- **D-29:** Portrait and landscape are tested with equal strictness.
- **D-30:** Display UAT must include daylight and low-light conditions.
- **D-31:** The live dash must communicate the core timing state within a one-second glance while mounted.
- **D-32:** Core display state includes current lap/elapsed lap time, delta or `--`, lap count or equivalent progress signal, last/best where relevant, GPS Ready/poor state, raw recording state, and timing active state.
- **D-33:** Phase 5.1 does not add safety restriction mechanisms. It does not introduce moving locks, speed-based UI locks, forced parking checks, or modal safety interruption flows.
- **D-34:** UI audit for Phase 5.1 does not grade whether the app restricts moving interaction. It grades timing-interface readability, state clarity, and data presentation.
- **D-35:** Existing project-level closed-course/private-track positioning is not changed by this phase. Changing or removing that requirement requires a separate requirements/project-scope change, not a Phase 5.1 implementation decision.

### UI Audit

- **D-36:** UI audit uses six equally weighted pillars: Domain Fit, Information Hierarchy, State Clarity, Motion/Glance Readability, Responsiveness / Device Fit, and Visual Polish / Accessibility.
- **D-37:** Each UI pillar must score at least 3/4. Any pillar at 1/4 or 2/4 blocks Go until fixed or re-audited.
- **D-38:** Domain Fit means the first screen must feel like a mounted-phone lap timing instrument. Current lap/elapsed time is the primary readout, delta is the second core readout, and GPS/timing state is clear. The UI must not drift into a generic fitness tracker, map recorder, or debug dashboard.
- **D-39:** Diagnostic data may exist, but it must not visually dominate the timing instrument.

### Code Audit and Hardening

- **D-40:** Code audit depth is deep across all Phase 1-5 core paths.
- **D-41:** Core audit scope includes architecture boundaries, lap engine, sector/crossing logic, replay, GPS Ready/quality/provider state, session lifecycle, storage, export, recovery, track profile/revision/direction/wrong-course handling, ghost reference/delta confidence, live timing UI state, and licensing/privacy risks tied to these paths.
- **D-42:** P0, P1, and P2 findings in the core paths must all be fixed before Go.
- **D-43:** P2 means all code-quality issues inside the Phase 5.1 core paths, not every possible repository-wide cleanup item.
- **D-44:** P3/info findings and unrelated future-feature cleanup may be recorded for backlog and do not block Go unless they affect core timing trustworthiness.
- **D-45:** Core architecture problems can trigger No-Go, especially lap engine dependency on UI/platform APIs, confused GPS/provider boundaries, storage risks that compromise timing evidence, or license/privacy issues in core paths.

### Evidence and Artifacts

- **D-46:** Phase 5.1 produces the complete audit package: `5.1-CODE-REVIEW.md`, `5.1-UAT.md`, `5.1-UI-REVIEW.md`, `5.1-FIELD-TEST-LOG.md`, and `5.1-GO-NOGO.md`.
- **D-47:** Each closed-course session must preserve session export, replay output, field notes, and video reference.
- **D-48:** Field notes use a minimal template: device, session ID/export file, lap count pass/fail, replay output/result, and video reference/result.
- **D-49:** Extra notes such as weather, mount position, lighting, temperature, battery, GPS Ready wait time, and abnormal events are optional and should be added when a session fails or behaves unusually.

### Go / Hardening Required / No-Go

- **D-50:** Phase 5.1 reports one of three outcomes: Go, Hardening Required, or No-Go.
- **D-51:** Go requires strict all-green results: core code audit P0/P1/P2 clear, five valid closed-course sessions with 100% lap count, deterministic replay pass, UI audit all six pillars >= 3/4, complete evidence package, and an explicit Android phone MVP Go conclusion.
- **D-52:** Hardening Required applies when blockers are clear and fixable but Go is not met: unresolved P0/P1/P2, UI pillar below 3, replay determinism failure with diagnosable cause, incomplete evidence, or fixable GPS Ready/raw recording/course-selection defects.
- **D-53:** No-Go applies to the current validation run if any valid closed-course session has incorrect lap count, if timing cannot be made trustworthy after replay/code diagnosis, if Ready cannot prevent untrustworthy timing, or if the audit finds core architecture defects that invalidate the phone timing path.
- **D-54:** Lack of closed-course access is not a product No-Go. It leaves Phase 5.1 pending/blocked until the required field evidence exists.

### Phase 6 and Phase 7 Unlocking

- **D-55:** A strict Phase 5.1 Go unlocks Phase 6.
- **D-56:** A strict Phase 5.1 Go also unlocks Phase 7, but Phase 7 starts from the validated Android phone timing state.
- **D-57:** Phase 7 scope after this gate is Android-source bridge first. The Android phone is the validated source of truth for the initial phone-to-glasses bridge work.
- **D-58:** iOS bridge behavior remains an explicit unvalidated risk and follow-up validation item. Phase 5.1 Android Go must not be written as iOS validation.

### the agent's Discretion

- Exact numeric GPS Ready thresholds for accuracy, sample age, and sample rate, as long as they are conservative and covered by replay or deterministic tests.
- Exact video annotation workflow, file naming, and comparison format for video-derived lap timing evidence.
- Exact UI audit rubric wording for each 1-4 score, as long as all six pillars remain and the 3/4 minimum is enforced.
- Exact severity taxonomy language, as long as P0/P1/P2 core-path findings block Go and P2 remains bounded to core paths.
- Exact format of the five Phase 5.1 artifacts, provided they preserve the decisions and evidence required above.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets

- Phase 3 storage/export and replay infrastructure should be used as the evidence path for field sessions.
- Phase 4 ghost reference and delta confidence behavior should be validated without adding map ghost or telemetry chart scope.
- Phase 5 course profile, revision, direction, and wrong-course override semantics define the course-selection state that Ready and UAT must verify.
- Existing deterministic replay fixtures and in-memory/file-backed storage patterns should be extended for Phase 5.1 validation rather than bypassed with manual-only checks.

### Established Patterns

- Shared domain logic owns lap timing, crossing, sector, ghost, and course semantics. UI and platform layers render state and forward actions.
- Algorithmic behavior must be testable through synthetic or recorded replay data.
- Local-first exports are the canonical debugging and evidence mechanism.
- Simulated/demo data remains isolated from real track best-lap and ghost calculations.
- The mounted-phone dash prioritizes large timing readouts and passive readability.

### Integration Points

- Ready state should aggregate GPS quality, sample freshness/rate, selected course/profile, confirmed start/finish, direction compatibility, and wrong-course preflight.
- Raw recording must remain possible when not Ready, but formal timing/lap/ghost state must not start in that mode.
- Replay tooling must compare deterministic algorithmic outputs across repeated runs.
- Field exports must be linkable from UAT and field-test logs to replay results and video evidence.
- UI audit should inspect the Drive/Timing surface, session state surfaces, and Review/export flows needed to support field evidence.

</code_context>

<specifics>
## Specific Ideas

- The field validation minimum is five valid closed-course sessions: three on the primary Android phone and two on a second available Android phone.
- The timing-evidence workaround for no professional timing hardware is fixed start/finish video plus deterministic replay.
- `<= 0.5s` lap-time difference versus readable video crossing is a target metric, not the hard Go blocker. Lap count correctness is the hard field blocker.
- A one-second glance should be enough to understand the live timing state.
- Phase 5.1 must not accidentally become an external-device or glasses validation phase.
- Phase 5.1 must not introduce new safety restriction mechanisms. The existing project positioning remains unless the project requirements are separately changed.

</specifics>
