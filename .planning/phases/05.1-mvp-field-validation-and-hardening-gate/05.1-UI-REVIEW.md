## Top 3 Priority Fixes

1. **Tokenize Visual Polish (Color/Type/Spacing)** — The UI relies on 30+ instances of hardcoded hex colors (`Color(0xFF...)`) and 100+ instances of inline `.sp` and `.dp` (e.g. `fontSize = 54.sp`, `padding(20.dp)`). This blocks accessibility scaling and breaks theming. Refactor to `MaterialTheme` semantic tokens.
2. **Harden Glance Readability (D-31/D-32)** — Hardcoded sizes (`42.sp` vs `54.sp`) risk clipping metric displays on foldable or edge-case screens. Convert manual sizing to semantic typographic bounds that respect viewport scaling, and ensure high-contrast colors for outdoor sunlight visibility.
3. **Refine State Clarity (GPS/Timing)** — Make the transition from "GPS Searching" to "GPS Ready" to "Timing Active" instantly recognizable via persistent status iconography rather than relying entirely on text labels that might be missed in a 1-second glance.

## Detailed Findings

### Pillar 1: Domain Fit (D-38) (3/4)
- **Assessment**: The UI successfully avoids feeling like a generic smartwatch dashboard or fitness tracker. The three-tab structure (Drive, Review, Settings) fits the mounted-phone use case perfectly.
- **Issue**: Some dialogs (e.g., `AppShell.kt:162` "You have a session that wasn't saved") feel overly conversational for a precision instrument.
- **Fix**: Standardize terminology to strict motorsport/timing instrument standards.

### Pillar 2: Information Hierarchy (3/4)
- **Assessment**: `DriveScreen.kt` correctly prioritizes primary telemetry. Speed and Delta are given massive focal weight (`DriveScreen.kt:767` font sizes).
- **Issue**: Because hierarchy is driven by hardcoded integers rather than semantic tags (`displayLarge`, `headlineMedium`), adding or modifying data fields requires manually re-balancing all `sp` and `dp` values.

### Pillar 3: State Clarity (3/4)
- **Assessment**: GPS permission and feed states are tied directly to visual feedback (`DriveScreen.kt:1048` color branches for `LocationFeedMode`).
- **Issue**: The "wrong course" blocking state or "waiting for start/finish line" state needs to be more explicitly bold to avoid a driver completing a lap before realizing timing wasn't active.

### Pillar 4: Motion / Glance Readability (D-31/D-32) (2/4)
- **Assessment**: `DeltaTone` mapping to Green/Orange/Gray (`DriveScreen.kt:998`) provides excellent peripheral color cues. 
- **Issue**: Relying on arbitrary `if (compact) 42.sp else 54.sp` without a fluid sizing container means long lap times or extreme speeds might clip off-screen during high-vibration driving, breaking the 1-second glance rule.

### Pillar 5: Responsiveness / Device Fit (3/4)
- **Assessment**: Landscape vs portrait is explicitly handled (`DriveScreen.kt:456` `isCompactLandscape`).
- **Issue**: Layout uses manual conditional padding (`spacedBy(if (isCompactLandscape) 8.dp else 10.dp)`). While it functions now, this hardcoded approach is fragile and doesn't utilize standard Compose WindowSizeClasses for proper device fitting.

### Pillar 6: Visual Polish / Accessibility (1/4)
- **Assessment**: This pillar represents a critical failure. The entire visual implementation sidesteps Android/iOS accessibility and theming systems.
- **Evidence**:
  - **Color**: 30+ hardcoded `Color(0xFF...)` instances (e.g., `DriveScreen.kt:314` `Color(0xFF101722)`).
  - **Typography**: 100+ inline `fontSize` declarations (e.g., `AppShell.kt:330` `28.sp`).
  - **Spacing**: 100+ inline `dp` values (e.g., `AppShell.kt:324` `padding(20.dp)`).
- **Fix**: Abstract all hardcoded values into `MaterialTheme.colorScheme`, `MaterialTheme.typography`, and a custom `Spacing` object.
