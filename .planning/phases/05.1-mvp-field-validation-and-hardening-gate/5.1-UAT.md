# 5.1 UAT — Android Field-Validation Protocol

**Phase:** 5.1 — MVP Field Validation and Hardening Gate
**Status:** Pending field evidence (D-54). This document is the executable protocol; Plan 07 carries the irreducible manual execution and fills `5.1-FIELD-TEST-LOG.md`.
**Scope gate:** Android phone-only (D-02). iOS, external GNSS, Meta glasses, watches, cloud, public-road use, advanced telemetry charts, and map ghost animation are explicitly OUT of scope for this gate.

This protocol is device-independent: anyone with two Android phones and a closed course can execute it without re-deriving thresholds or acceptance rules. It exercises both the **live timing path** (Drive/Ready/lap timing) and the **saved-session Review + JSON/GPX export path**, and ties every field anomaly to deterministic replay before any root cause is assigned.

---

## 0. Outcome semantics (how this UAT feeds Go/No-Go)

| Outcome | Trigger |
|---------|---------|
| **No-Go (this run)** | Any valid closed-course session produces an incorrect lap count (D-19/D-20), or timing cannot be made trustworthy after replay/code diagnosis, or Ready cannot prevent untrustworthy timing. |
| **Hardening Required** | A fixable, diagnosable defect (Ready/raw-recording/course-selection/replay drift) blocks Go but is not a lap-count failure. |
| **Go** | Five valid sessions at 100% lap count, deterministic replay pass, Review/export smoke pass, and the UI/code audits clear. |
| **Pending (not No-Go)** | Closed-course access or a second device is unavailable. The phase stays pending/blocked until field evidence exists — lack of access is never a product No-Go (D-54). |

- **Android-only** (D-02). Do not record iOS, external GNSS, glasses, or watch results as field evidence (D-03).

  - **Secondary phone** — any other available Android phone. It need NOT be forced low-end, old, a specific brand, chipset, or screen size (D-06).

## 4. Mounted-display UAT (one-second-glance)

Tested with **equal strictness in portrait and landscape** (D-29) and in **both daylight and low-light** (D-30). Judge each of the core state elements for legibility within a one-second glance while the phone is mounted and the vehicle is in motion (D-31).

**Core display state that must be glanceable in ≤ 1 second** (D-32):

1. Current lap / elapsed lap time (primary readout).
2. Delta to best, or `--` when unavailable (second core readout).
3. Lap count or an equivalent progress signal.
4. Last lap and best lap, where relevant.
5. GPS Ready / poor state.
6. Raw-recording state (active / not).
7. Timing-active state (timing running / not).

| Condition | Portrait | Landscape |
|-----------|----------|-----------|
| Daylight (direct sun on glass) | ☐ all 7 legible ≤1s | ☐ all 7 legible ≤1s |
| Low-light / dusk | ☐ all 7 legible ≤1s | ☐ all 7 legible ≤1s |

**Pass:** all seven elements legible within one second in all four condition cells.
**Note (D-33/D-34):** This phase grades readability/state-clarity/data-presentation ONLY. Do NOT add or grade moving-locks, speed-based UI locks, forced parking checks, or modal safety interruptions. Diagnostic data may exist but must not visually dominate the timing instrument (D-39).

## 5. Ready protocol (Ready-before-timing rule)

**A session is valid for accuracy and Go/No-Go ONLY if the app itself showed the required Ready state before timing began (D-13).** If timing started while not Ready, the run is invalid as evidence regardless of how the laps looked.

**Chosen conservative Ready thresholds (transcribed verbatim from Plan 03, D-14 discretion):**

| Ready input | Threshold | Source / rationale |
|-------------|-----------|--------------------|
| Horizontal accuracy | **≤ 25.0 m** (`maxHorizontalAccuracyMeters = 25.0`) | Engine degraded-accuracy line (`GpsQualitySummary.DEFAULT_DEGRADED_ACCURACY_METERS`), stricter than preflight's 100 m course-distance gate. |
| Fix freshness (sample age) | **≤ 15000 ms** (`maxFixAgeMillis = 15_000`) | Wrong-course preflight freshness window (15 s). |
| Sample rate / cadence | **≥ 0.9 Hz** (`minSampleRateHz = 0.9`) | Observed real Fused-Location delivery floor (RESEARCH Nyquist note: requested ~10 Hz, real delivery often hovers around 1 Hz). Below this a start/finish crossing cannot be trustworthily reconstructed. |

Ready additionally requires (aggregate gate, D-14): a valid GPS fix present, a selected course/profile, a confirmed start/finish line, and course-direction compatibility. **Motion, speed, and heading are NOT Ready inputs** — a stationary good fix can be Ready (D-15).

**Rules:**
- If the app is **not Ready**, the user MAY start **raw GPS recording for diagnosis**, but the app must NOT enter formal lap timing or ghost timing for that recording (D-16). Raw recording while not Ready does NOT count toward the five valid sessions (D-17).
- A **clearly wrong-course preflight** blocks Ready (consistent with Phase 5). The explicit **"Still use this track" override** remains allowed and must be recorded as session evidence visible in Review/export (D-18). The override runs before the Ready gate and intentionally bypasses it; if used, note it in the field log.
- Timing/speed/delta must be suppressed or degraded honestly when input data cannot support a trustworthy result (D-24).

**Per-session Ready checklist (record in field log):**
- ☐ App showed Ready before timing start (accuracy ≤25 m, age ≤15 s, rate ≥0.9 Hz, course/start-finish/direction satisfied).
- ☐ If override used, "Still use this track" evidence is visible in Review/export.

## 7. Acceptance metrics

| Metric | Standard | Type |
|--------|----------|------|
| **Lap count** | **100% correct** vs manual/video tally for every valid session. | **HARD blocker** (D-19). Any valid session with an incorrect lap count is a No-Go for that run; fix and rerun affected validation before Phase 6/7 (D-20). |
| Lap time vs readable video crossing | **≤ 0.5 s** difference for laps where the video clearly shows the start/finish crossing. | **Target metric**, not an automatic Phase 6 blocker (D-21/D-22). |
| Unreadable video crossings | Excluded from timing-accuracy conclusions; may still support stability/storage/replay/UI/lap-count evidence (D-23). | Exclusion rule. |

There is no professional-timing-device requirement; **video reference plus deterministic replay is the accepted workaround** (D-22).

## 8. Replay-diagnosis-first protocol

**Every field failure or timing anomaly MUST be decoded and replayed through `SessionReplayDecoder` before a root cause is assigned (D-28).** No anomaly is attributed from memory or live observation alone.

**Replay entry point (Plan 01):** `SessionReplayDecoder.decodeForReplay(exportedBytes, config)` decodes the exported session JSON with the same `FileSessionStore.canonicalJson` `JsonExportService` encodes with, rebuilds the course via `courseFromTrack`, and runs `ReplayRunner`. It returns a typed `ReplayDecodeResult.Decoded / Corrupt / NoCourse` (bad/truncated/degenerate bytes are data, never an exception).

**Diagnosis loop for any anomaly:**
1. Export the session JSON from the device.
2. `decodeForReplay(...)` → confirm `Decoded` (a `Corrupt`/`NoCourse` result is itself the finding).
3. Re-run the decode **twice** and confirm byte-identical algorithmic output (lap count, lap times, sector times, ghost delta state transitions, course/profile/direction decisions). Only display formatting/rounding may differ (D-25/D-26/D-27).
4. Assign root cause to exactly one of: **data quality / line placement / interpolation / Ready classification / course selection / implementation logic** (D-28).
5. **Each field export becomes a permanent replay fixture** that must thereafter reproduce identical output (D-25); link it from `5.1-FIELD-TEST-LOG.md`.

**Replay acceptance:** `decodeForReplay` returns `Decoded`, replays deterministically across repeated runs, and its lap count matches the manual/video tally. A replay-determinism failure with a diagnosable cause is Hardening Required (D-52); an undiagnosable trustworthiness failure is No-Go (D-53).

## 9. Review + JSON/GPX export smoke (per valid session, on device)

After recording a valid session, exercise the saved-session review and export paths — not just the live-timing path. Run on a real device (emulator does not count).

| # | Check | Requirement | Concrete pass/fail acceptance |
|---|-------|-------------|-------------------------------|
| R1 | Open the **Review tab**; confirm the saved session is listed with its **lap list, best lap, total duration, and GPS quality summary**. | SESS-02 | **PASS:** Review tab lists the saved session and shows all four (lap list, best lap, total duration, GPS quality). **FAIL:** session missing or any of the four absent. |
| R2 | Confirm the **simple track trace** renders for that session. | SESS-03 | **PASS:** the track trace draws the session path. **FAIL:** blank/empty/erroring trace. |
| R3 | Save a completed lap as the **reference/best lap** and confirm it is retained. | GHOST-01 | **PASS:** the lap is saved as reference and persists (still present on reopen). **FAIL:** save fails or is not retained. |
| R4 | **Export the session as JSON**, then re-replay it. | SESS-04/SESS-05 | **PASS:** JSON export re-replays through `SessionReplayDecoder` to identical, lap-count-correct output. **FAIL:** decode returns `Corrupt`/`NoCourse`, drifts across runs, or lap count mismatches. |
| R5 | **Export the session as GPX** (or another common interchange format). | SESS-05 | **PASS:** the GPX file is a non-empty `.gpx` that re-imports/parses as a well-formed track with points. **FAIL:** empty file, malformed XML, or no track points. |

**Smoke acceptance summary:** Review tab lists the saved session with lap list/best lap/duration/GPS quality and a rendering track trace; a completed lap saves and persists as the reference lap; the JSON export re-replays to identical output; the GPX export is a non-empty file that re-imports/parses with track points.
