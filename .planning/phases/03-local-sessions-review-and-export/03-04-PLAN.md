<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| UI -> timing session controller | User starts/stops timing and chooses Save/Discard/Resume. |
| Provider samples -> LapEngine recorder | GPS samples produce timing events and persisted raw traces. |
| Draft payloads -> storage/load recovery | Unfinished JSON payloads are loaded on app launch. |
| Simulated data -> ghost candidate derivation | Demo sessions are saved but must not affect real best-lap candidates. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-11 | Tampering/DoS | draft JSON load | mitigate | Reuse typed schema validation and corrupt/quarantine handling from `FileSessionStore`; recovery prompt must handle corrupt/missing drafts without crashing. |
| T-03-12 | Repudiation | Save/Discard actions | mitigate | Model explicit draft states and require Save/Discard actions after Stop; formal Review history rows only appear after Save. |
| T-03-13 | Information Disclosure | timing session payloads | mitigate | Store raw samples and laps in app-private canonical payload files; no export/share action occurs in this plan. |
| T-03-14 | Integrity | `GhostCandidate` | mitigate | Filter simulated/demo sessions out of real per-track fastest-lap candidate calculations; tests assert D-43. |
| T-03-15 | Safety | fullscreen Drive UI | mitigate | Keep passive mounted-phone UI, large timing fields, visible GPS/source state, and closed-course safety copy; no public-road racing positioning. |
